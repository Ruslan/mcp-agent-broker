# План 0.0.8: Опциональный API-ключ

## Цель
Добавить опциональную защиту API через ключ из `.env`. Если ключ задан, клиент обязан передавать его в каждом запросе. Если ключ не задан, сервер работает как сейчас, без дополнительной проверки.

## Решение
- Использовать опциональную переменную окружения `API_KEY`.
- Передавать ключ через заголовок `Authorization: Bearer <key>`.
- Проверять ключ в общей точке обработки HTTP API, чтобы не дублировать логику по маршрутам.
- При отсутствии или неверном ключе возвращать `401 Unauthorized` с короткой понятной ошибкой.

## Пошаговый план реализации
1. Добавить чтение опциональной переменной `API_KEY` из `.env` в конфигурацию сервера.
2. Найти общую точку входа HTTP API и встроить туда единую проверку API-ключа.
3. Реализовать поведение:
   - если `API_KEY` не задан, запросы проходят без изменений;
   - если `API_KEY` задан, сервер требует заголовок `Authorization: Bearer <key>`.
4. Сравнивать переданный ключ со значением из конфигурации и отклонять невалидные запросы с `401 Unauthorized`.
5. Проверить, нужны ли исключения для служебных эндпоинтов, например `health` или внутренних маршрутов. Если такие эндпоинты есть, явно зафиксировать их поведение.
6. Обновить `.env.example` и документацию: описать, что `API_KEY` опционален, как включить защиту и как клиент должен передавать ключ.
7. Добавить тесты на основные сценарии:
   - `API_KEY` не задан, запросы проходят без ключа;
   - `API_KEY` задан, запрос без ключа получает `401`;
   - `API_KEY` задан, запрос с неверным ключом получает `401`;
   - `API_KEY` задан, запрос с правильным ключом проходит.

## Примечания
- Не логировать значение API-ключа в ошибках, отладочных сообщениях и трассировках.
- Поведение без `API_KEY` должно остаться полностью совместимым с текущим режимом работы.
